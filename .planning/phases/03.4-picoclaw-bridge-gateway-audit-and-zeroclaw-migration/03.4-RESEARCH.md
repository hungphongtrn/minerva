# Phase 03.4: Picoclaw bridge gateway audit and Zeroclaw migration - Research

**Researched:** 2026-03-05
**Domain:** Runtime Gateway Integration, Bridge Protocol Audit, Conditional Migration
**Confidence:** MEDIUM (Picoclaw well understood; Zeroclaw is new/unspecified system)

## Summary

This phase addresses urgent runtime gateway stability needs by auditing the current Picoclaw bridge gateway implementation and conditionally migrating to Zeroclaw if Picoclaw fails to meet the minimum bar.

The current Minerva implementation has a working bridge service (`PicoclawBridgeService`) that executes HTTP requests against in-sandbox Picoclaw gateways at port 18790. However, recent debug evidence shows multiple failure modes:

1. **DB/Orphaning issues**: Sandboxes created in Daytona without corresponding DB rows (resolved by early checkpoint commits)
2. **Bridge startup gaps**: Daytona sandboxes created without Picoclaw runtime bridge listener on port 18790 (partially resolved with startup hooks)
3. **Gateway URL drift**: Incorrect DNS patterns from hardcoded URL construction (resolved via preview-link authority)
4. **Local compose simulation**: No real gateway for local development (by design, but blocks full testing)

**Primary recommendation:** Audit Picoclaw gateway against "full run experience" criteria (streaming/events, tool calls, session continuity) in real Daytona sandboxes. If gaps exist, implement Zeroclaw as a clean replacement with big-bang cutover (no rollback needed). Remove all Picoclaw artifacts if migration proceeds.

### Pattern 1: Gateway Service Abstraction
**What:** Abstract base defining gateway contract; concrete implementations for Picoclaw/Zeroclaw
**When to use:** If audit reveals Picoclaw and Zeroclaw share common patterns

**Example:**

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class GatewayResult:
    success: bool
    output: Optional[Dict[str, Any]] = None
    error: Optional[GatewayError] = None

class BaseGatewayService(ABC):

        pass
    
    @abstractmethod

        self,

        context: ExecutionContext,
        auth: AuthBundle
    ) -> GatewayResult:
        """Execute message via gateway."""
        pass
```

### Pattern 2: Fail-Closed Execution Flow
**What:** Health check → Auth validation → Execute with bounded retries
**When to use:** All gateway integrations (current pattern in PicoclawBridgeService)

**Verified pattern from codebase:**
```python
# Source: src/services/picoclaw_bridge_service.py lines 336-486
async def execute(self, sandbox_url: str, message: str, ...) -> BridgeResult:
    # Step 1: Fail-closed auth check
    if not token_bundle or not token_bundle.current:
        return BridgeResult(success=False, error=BridgeError(...))
    
    # Step 2: Health polling (fail-closed)
    health = await self.poll_health(sandbox_url, token_bundle)
    if not health.healthy:
        return BridgeResult(success=False, error=BridgeError(...))
    
    # Step 3: Execute with bounded retries
    for attempt in range(self.execute_retries + 1):
        result = await self._try_execute(...)
        if result.success:
            return result
        if not self._is_recoverable(result.error):
            return result  # Fail fast
    
    return BridgeResult(success=False, error=last_error)
```

### Pattern 3: Token Bundle with Grace Period
**What:** Sandbox-scoped auth with current + previous token rotation support
**When to use:** When gateway requires per-sandbox auth with rotation

**Verified pattern:**
```python
# Source: src/services/picoclaw_bridge_service.py lines 68-114
@dataclass
class BridgeTokenBundle:
    current: str
    previous: Optional[str] = None
    previous_expires_at: Optional[datetime] = None

    def get_effective_token(self, attempt: int = 0) -> str:
        if attempt == 0:
            return self.current
        if self.is_grace_token_valid():
            return self.previous
        return self.current
```

### Anti-Patterns to Avoid
- **Synthetic URL construction**: Never fabricate gateway URLs; use authoritative sources only (Daytona preview links)
- **Request-transaction durability**: Never rely on request-level transaction for external resources (Daytona workspace can survive DB rollback)
- **Silent hydration failures**: Always validate process.exec results for non-zero exit codes
- **Unbounded recovery loops**: Cap recovery attempts (current: MAX_RECOVERY_ATTEMPTS = 3)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Gateway health checking | Custom polling logic | `PicoclawBridgeService.poll_health()` with retries/backoff | Already implements bounded retry, exponential backoff, typed errors |
| HTTP client management | Raw `aiohttp` or `requests` | `httpx.AsyncClient` with timeouts | Used throughout; supports both sync and async; proper timeout handling |
| Token rotation | Naive token replacement | `BridgeTokenBundle` with grace period | Handles cutover periods without auth failures |
| Gateway URL derivation | String concatenation | Daytona preview-link APIs | `info.get_preview_link()` from Daytona SDK |
| Error typing | String error messages | `BridgeErrorType` enum with remediation | Allows programmatic error handling and remediation guidance |
| Session key generation | Random UUIDs | `_generate_session_key()` with scope | Ensures deterministic routing and session continuity |

**Key insight:** The current `PicoclawBridgeService` already implements production-grade patterns (fail-closed, bounded retry, token rotation). If migrating to Zeroclaw, replicate these patterns rather than inventing new ones.

## Common Pitfalls

### Pitfall 1: Bridge Runtime Startup Gap
**What goes wrong:** Daytona workspace shows STARTED but no listener on port 18790
**Why it happens:** Daytona provisioning completes before Picoclaw runtime starts bridge process
**How to avoid:** 
- Implement explicit bridge startup in provisioning (done in recent fixes)
- Verify listener readiness before marking sandbox ACTIVE
- Add hydration retry counter to prevent infinite loops
**Warning signs:** Health check timeouts immediately after provisioning; empty bridge responses

### Pitfall 2: Orphaned Daytona Workspaces
**What goes wrong:** Daytona workspace exists but no `sandbox_instances` DB row
**Why it happens:** Request-level transaction rollback after Daytona create but before DB commit
**How to avoid:** 
- Early checkpoint commit before provider provisioning (implemented)
- Best-effort orphan reconciliation with workspace_id labels
**Warning signs:** Gateway URL DNS failures; routing errors for workspace that "should exist"

### Pitfall 3: Local Compose Simulation vs Reality
**What goes wrong:** Local development works but Daytona fails, or vice versa
**Why it happens:** Local compose is simulation (no real gateway); Daytona is real infrastructure
**How to avoid:** 
- Detect local compose URLs (`http://local-sandbox-*`) and fail fast with clear message
- Test full execution only in Daytona infrastructure
**Warning signs:** `bridge_transport_error` with local compose; success in local but failure in CI

### Pitfall 4: Session Key Collisions
**What goes wrong:** Different users share session; or same user gets different sessions unexpectedly
**Why it happens:** Incorrect session key scoping or missing `session_id` continuity
**How to avoid:** 
- Use `minerva:{workspace_id}:{pack_scope}:{continuity_key}` format
- Forward `session_id` for explicit continuity
- Guest sessions must be ephemeral (unique per request)
**Warning signs:** Session state bleeding between users; lost conversation context

### Pitfall 5: Gateway URL Drift
**What goes wrong:** DNS failures connecting to `gateway-{id}.daytona.run`
**Why it happens:** Hardcoded hostname construction doesn't match Daytona preview URL format
**How to avoid:** 
- Use `SandboxInfo.get_preview_link()` from Daytona SDK
- Never prepend `gateway-` to preview hostnames
**Warning signs:** `gateway-*` DNS failures; connection refused to synthetic URLs

## Code Examples

### Verified Pattern: Bridge Execute with Recovery
```python
# Source: src/services/run_service.py lines 970-1101
async def _execute_via_bridge(
    self,
    routing: RunRoutingResult,
    message: str,
    is_guest: bool,
    session: Session,
    session_id: Optional[str] = None,
    sender_id: Optional[str] = None,
) -> BridgeResult:
    MAX_RECOVERY_ATTEMPTS = 3
    
    for attempt in range(MAX_RECOVERY_ATTEMPTS):
        # Get authoritative URL
        sandbox_url = self._get_authoritative_sandbox_url(routing)
        if not sandbox_url:
            return BridgeResult(success=False, error=BridgeError(...))
        
        # Resolve auth tokens
        token_bundle = self._resolve_bridge_tokens(routing, session)
        if not token_bundle or not token_bundle.current:
            return BridgeResult(success=False, error=BridgeError(...))
        
        # Execute
        bridge_service = PicoclawBridgeService()
        result = await bridge_service.execute(...)
        
        if result.success:
            return result
        
        # Check recoverable
        if not self._is_recoverable_bridge_error(result.error):
            return result  # Fail fast
        
        # Attempt recovery (reprovision)
        if attempt < MAX_RECOVERY_ATTEMPTS - 1:
            routing = await self._recover_routing_target(routing, session)
    
    return BridgeResult(success=False, error=BridgeError(...))
```

### Verified Pattern: Sandbox-Scoped Token Resolution
```python
# Source: src/services/run_service.py lines 1149-1192
def _resolve_bridge_tokens(
    self, routing: RunRoutingResult, session: Session
) -> Optional[BridgeTokenBundle]:
    if not routing.sandbox_id:
        return None
    
    repo = SandboxInstanceRepository(session)
    token_data = repo.resolve_bridge_tokens(sandbox_uuid)
    
    if not token_data or not token_data.get("current"):
        return None
    
    return BridgeTokenBundle(
        current=token_data["current"],
        previous=token_data.get("previous"),
        previous_expires_at=token_data.get("previous_expires_at"),
    )
```

### Verified Pattern: Session Key Generation
```python
# Source: src/services/run_service.py lines 936-968
def _generate_session_key(
    self,
    workspace_id: Optional[str],
    agent_pack_id: Optional[str],
    run_id: str,
    is_guest: bool,
    session_id: Optional[str] = None,
) -> str:
    if is_guest:
        return f"minerva:guest:{run_id}"
    
    pack_scope = agent_pack_id or "default"
    continuity_key = session_id if session_id else run_id
    return f"minerva:{workspace_id}:{pack_scope}:{continuity_key}"
```
