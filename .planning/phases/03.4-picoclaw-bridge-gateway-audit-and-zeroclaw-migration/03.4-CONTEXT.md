# Phase 03.4: picoclaw-bridge-gateway-audit-and-zeroclaw-migration - Context

**Gathered:** 2026-03-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Audit the current Picoclaw bridge gateway path to determine whether it provides a working gateway for Minerva's full run experience in Daytona, and (if Picoclaw does not meet the minimum bar) migrate Minerva's runtime integration to Zeroclaw.

This phase clarifies HOW we audit and migrate within this boundary. New capabilities belong in other phases.

</domain>

<decisions>
## Implementation Decisions

### Cutover strategy
- Current project state: still under development; no external users yet.
- Cutover approach: big-bang.
- Rollback: not required.
- Old Picoclaw path availability after cutover: can be removed immediately.
- Routing stickiness: not specified (question was unclear); treat as not required given big-bang/no-rollback.

### Compatibility contract
- Compatibility posture: anything can change to match Zeroclaw (API/event/error contracts are allowed to change).
- Callers: update all in-repo consumers/tests as needed in the same phase.
- Session/thread continuity: desired and prioritized, but not a hard blocker (aspirational/best-effort).

### Audit scope (Picoclaw gateway)
- Purpose: documentation/evidence audit (not a "fix Picoclaw" phase).
- Environment: audit must be valid in Daytona sandboxes.
- What we are validating: whether Picoclaw exposes a working gateway for Minerva's use case.
- Minimum bar: full run experience (streaming/events, tool calls, and session continuity expectations) in Daytona.
- Artifacts: automated tests + a short written report/checklist.
- If audit finds risks/issues: no requirement to fix Picoclaw; the audit informs whether Zeroclaw should become primary.

### Safety + observability
- Visibility: high internal visibility (structured logs and per-run auditability/correlation).
- Client exposure: routing choice (Zeroclaw vs Picoclaw) stays internal-only.
- Failure posture: fail fast with clear remediation if Zeroclaw path fails.
- Migration safety: require a green/soak window before considering the migration safe (exact definition left open).

### Zeroclaw contract
- Role: full replacement when used (Zeroclaw becomes the primary gateway/runtime integration path).
- Readiness bar: full run experience.
- Environment priority: Daytona-first.
- Relationship to Picoclaw audit: conditional migration.
  - If Picoclaw meets the minimum bar, keeping Picoclaw is acceptable.
  - If Picoclaw does not meet the bar, proceed with Zeroclaw as the primary path.

### Deprecation cleanup (conditional on migrating)
- Old Picoclaw gateway integration: delete in the same phase.
- Fallback: no fallback kept.
- Naming/config/docs: rename external config/env/docs from Picoclaw/bridge to Zeroclaw.
- Repo hygiene: update relevant tests/docs and remove obsolete Picoclaw-specific artifacts.

### OpenCode's Discretion
- Define the concrete soak window (duration, run volume, and which metrics/thresholds qualify as "green").
- Any remaining details of rollout routing consistency (stickiness) given big-bang/no-rollback.

</decisions>

- Zeroclaw is allowed to reshape streaming and error semantics; internal callers can be updated to match.
