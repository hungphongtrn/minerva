# Phase 2.1: Bridge Agent Pack Sandbox to Picoclaw Runtime - Context

**Gathered:** 2026-02-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Urgent bridge work from sandbox-mounted agent packs to in-sandbox Picoclaw runtime invocation and request execution.

This phase clarifies how Minerva invokes Picoclaw runtime inside the provisioned sandbox and returns execution results through existing run APIs. New capabilities (streaming protocols, broader channel integrations) are out of scope.

</domain>

<decisions>
## Implementation Decisions

### Runtime invocation mode
- Use Picoclaw `gateway` mode inside the sandbox (not one-shot `agent` mode).
- Orchestrator performs readiness polling against `/health` before routing execution requests.
- Orchestrator owns runtime lifecycle; if Gateway crashes, respawn the sandbox.

### Communication protocol
- Use synchronous HTTP bridge semantics for orchestrator <-> in-sandbox runtime communication.
- Require bearer-token authentication for bridge calls, injected per sandbox via environment variables.
- External payload contract is Minerva envelope (not raw Picoclaw `InboundMessage` payloads).
- Bridge returns a single final response for this phase (no chunk streaming).
- Runtime bridge failures reuse existing typed error/remediation contract style from current run routing APIs.

### Configuration generation
- Generate `config.json` per sandbox/workspace instance (deterministic, runtime-generated).
- Pass sensitive credentials via environment variables only; do not persist secrets in pack files or DB records.
- Keep configuration isolated per workspace sandbox, not shared globally.
- Enable bridge-only channel behavior in this phase; keep public chat channels disabled.

### Agent pack to Picoclaw workspace mapping
- Materialize pack content as snapshot copy/sync into sandbox workspace (provider-specific implementation), not live host bind semantics.
- Map pack root as Picoclaw workspace root so `AGENTS.md`, `SOUL.md`, `IDENTITY.md`, and `skills/` are directly discoverable.
- Keep native Picoclaw skill precedence: workspace skills -> global skills -> builtin skills.
- If source changes after registration/provisioning, fail closed as stale and require re-register/re-provision.

### Request/response execution model
- Keep `/runs` request path synchronous for this phase: wait for runtime completion in same API call.
- On execution timeout, return retryable typed runtime/infrastructure error with remediation guidance.
- Success payload must include existing run metadata plus final assistant output text.
- Streaming/token-by-token output is deferred to a later phase.

### Session isolation and continuity
- Session scope is `workspace + agent_pack` (prevent cross-pack context bleed).
- Session continuity is required across sandbox respawns (not only within a single active process lifetime).
- Guest sessions remain ephemeral per request.
- Any cross-workspace session reuse attempt is rejected fail-closed.

### OpenCode's Discretion
- Exact timeout budget and retry-backoff values for synchronous runtime execution.
- Concrete field naming/versioning for bridge envelope internals while preserving decided semantics.
- Concrete mechanism for cross-respawn session continuity (as long as workspace+pack isolation and fail-closed behavior are preserved).

</decisions>

<specifics>
## Specific Ideas

- Prioritize Picoclaw features already available in `references/picoclaw`, especially Gateway runtime operation and health/readiness checks.
- Keep Phase 2 control-plane behavior style: deterministic outcomes, fail-closed routing, and profile-portable semantics.

```mermaid
flowchart LR
  A[Client POST /runs] --> B[RunService + Lifecycle Routing]
  B --> C[Sandbox Selected or Provisioned]
  C --> D[Picoclaw Gateway /health ready]
  D --> E[HTTP Bridge Invocation]
  E --> F[Picoclaw Runtime Executes Agent Pack]
  F --> G[Final Response + Typed Errors]
```

</specifics>

<deferred>
## Deferred Ideas

- Token streaming and incremental event delivery are deferred to later streaming-focused phase work.
- Async callback/job mode is deferred; this phase locks synchronous execution semantics.
- Enabling public chat channels (Telegram/Discord/etc.) in sandbox runtime is deferred.

</deferred>
