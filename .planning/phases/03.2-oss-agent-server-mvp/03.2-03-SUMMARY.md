## Decisions Made

### Volume Naming and Immutability

**Decision:** Use deterministic volume naming `agent-pack-{pack_id}-{digest}` to ensure immutable volumes.

**Rationale:** 
- Same pack+digest always maps to same volume (deterministic)
- Digest changes create new volume (immutable)
- Old volumes remain for running sandboxes (graceful rollout)
- Multiple sandboxes can mount same volume (shared, read-only)

### Disposable Sandbox Pattern

**Decision:** Use disposable sandboxes for pack file upload with guaranteed cleanup in `finally` blocks.

**Rationale:**
- Ensures no resource leaks even on upload errors
- Clean separation between sync operations and production sandboxes
- Allows bulk upload via `sandbox.fs.upload_files()` for efficiency
- Follows fail-safe cleanup patterns

### Real File API Identity Verification

**Decision:** Replace simulated identity verification with real file API checks.

**Rationale:**
- Fail-closed security: must verify actual file existence
- Checks `/workspace/pack/AGENT.md`, `/workspace/pack/SOUL.md`, `/workspace/pack/IDENTITY.md`, `/workspace/pack/skills/`
- Hard gate: sandbox not ready if identity files missing
- Uses Daytona SDK `sandbox.fs` API for real checks

### Config Isolation

**Decision:** Write per-sandbox Picoclaw config.json to `~/.picoclaw/config.json` outside shared volume.

**Rationale:**
- Prevents config pollution across sandboxes
- Each sandbox gets unique bridge auth token and LLM settings
- Shared volume remains read-only pack files only
- Follows Picoclaw runtime invariants (static identity files mounted at creation)

### Register Command Error Handling

**Decision:** Block on critical failures with distinct exit codes.

**Rationale:**
- Exit 1: Validation errors (pack structure invalid)
- Exit 2: Database errors (persistence failed)
- Exit 3: Volume sync errors (Daytona unavailable)
- Clear error messages guide operator remediation
- Non-zero exit enables CI/CD integration
