# Phase 3.3: Close pack-mount isolation and identity-collision gaps - Research

**Researched:** 2026-03-03
**Domain:** Identity Management and Infrastructure Isolation
**Confidence:** HIGH

## Summary

This research identifies the technical implementation path for closing two critical security and architectural gaps: identity collision between developers and end-users, and the mixing of static pack data with dynamic runtime data in sandbox mounts.

The core solution involves decoupling end-user identities into a dedicated `external_identities` table scoped by `workspace_id`, and enforcing a strict isolation boundary between the read-only pack volume (`/workspace/pack`) and a new writable workspace directory (`/home/daytona/workspace`) using symlinks for identity files.

**Primary recommendation:** Use `sandbox.process.exec` to create symlinks during provisioning and update the Picoclaw bridge to forward raw external user IDs as `sender_id`.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- New `external_identities` table — completely separate from the developer `users` table. No shared rows, no collision possible.
- End-users are scoped to the developer's workspace (not their own). The developer registers packs in their workspace; all end-users use that workspace's packs.
- Uniqueness is per-workspace: composite key on (workspace_id, external_user_id). Two different deployments can both have an end-user called `user-123` without collision.
- Developer workspace resolved via explicit env var `MINERVA_WORKSPACE_ID`. Developer runs `minerva register` which prints the workspace ID, then sets it in config.
- If `MINERVA_WORKSPACE_ID` is not set and a `/runs` request arrives, fail closed with 403 and clear guidance telling the developer to set the env var.
- Minimal record in `external_identities`: workspace_id + external_user_id + created_at. No activity metadata — observability comes from logs/metrics.
- Auto-create on first request: any new `X-User-ID` value from the developer's gateway automatically upserts into `external_identities`. No pre-registration needed.
- `GUEST_ID` env var designates which `X-User-ID` value is treated as guest. Guest gets ephemeral sandbox, no `external_identities` row, no session continuity.
- Forward the raw `external_user_id` (from `X-User-ID`) as `sender_id` to Picoclaw via the bridge. No hashing or transformation — the developer controls the format, minerva passes it through.
- Forward `session_id` (from `X-Session-ID`) alongside `sender_id` in the bridge payload. Picoclaw uses both for conversation thread scoping within the sandbox.
- Guest requests (X-User-ID matches GUEST_ID) use generic `'guest'` as sender_id — the agent knows it's a guest but has no unique identifier.
- Separate paths: pack volume stays mounted at `/workspace/pack` (static, shared, read-only identity files). Picoclaw workspace directory moves to `/home/daytona/workspace` (dynamic, per-sandbox, writable runtime data).
- Orchestrator creates symlinks via Daytona file API during provisioning: AGENT.md, SOUL.md, IDENTITY.md, skills/ symlinked from `/workspace/pack/` into `/home/daytona/workspace/`.
- Picoclaw `config.json` workspace field updated to point to `/home/daytona/workspace` (currently points to `/workspace/pack`).
- Checkpoint captures only `/home/daytona/workspace` (memory/, sessions/, cron/) — never the pack volume. On restore, symlinks are recreated from the current pack volume.
- Full parity now — same isolation pattern as Daytona in this phase. Developers see identical behavior locally and in production.
- Docker bind mounts: pack files bind-mounted read-only at `/workspace/pack`, workspace dir is a local writable path at `/home/daytona/workspace`. Symlinks created the same way as Daytona.
- Full identity forwarding parity: local_compose also forwards `external_user_id` as `sender_id` and `session_id` to the bridge (same as Daytona path).
- Base provider contract (abstract class in `base.py`) codifies the isolation contract. Both providers implement the same interface: mount pack at X, workspace at Y, symlink identity files, write config outside pack.
- Clean cut: new `external_identities` table added via schema-only Alembic migration. Old `User` rows created by the previous `external_identity` code remain but are ignored by the new code. No data migration needed — orphaned rows are harmless.
- Active sandboxes with old mount layout: teardown all existing sandboxes on upgrade. Next request triggers fresh provisioning with the new mount layout. Brief downtime for end-users is acceptable.
- Identity file verification checks at the workspace path (`/home/daytona/workspace/AGENT.md` etc.), not the pack volume path. This confirms the full chain works: volume mounted + symlinks created + files readable.
- If verification at the workspace path fails (symlink broken, file missing), fail the sandbox as unhealthy and trigger reprovision. No recovery attempts — broken symlinks mean provisioning failed.
- Validate `MINERVA_WORKSPACE_ID` at startup (preflight): check env var is set, workspace exists in DB, and workspace has at least one registered agent pack. Server refuses to start if any check fails.
- If workspace exists but has no packs, preflight fails with: "Workspace has no registered agent packs. Run `minerva register` first."
- OSS deployments serve exactly one agent. Single pack per workspace model. If multiple packs exist in the workspace, preflight warns or fails. Matches the "one server = one agent" OSS model.
- Sandbox keyed on (workspace_id, external_user_id). Each end-user gets their own sandbox within the developer's workspace.
- Lookup: find active healthy sandbox for this workspace + external_user_id combo. No longer keyed on workspace ownership.
- Both paths stay active in the same server. `/api/v1/` is for developer management (register packs, manage keys, check status) using API key auth. `/runs` is for end-user message execution using X-User-ID passthrough. Different auth, different purpose, no conflict.

### Pattern 1: Symbolic Link Isolation
**What:** Creating symlinks from a Read-Only volume to a Read-Write workspace.
**When to use:** When a third-party application (Picoclaw) expects a single directory for both identity (static) and data (dynamic).
**Example:**
```python
# Source: Daytona SDK docs / Research
await sandbox.process.exec("ln -s /workspace/pack/AGENT.md /home/daytona/workspace/AGENT.md")
await sandbox.process.exec("ln -s /workspace/pack/skills /home/daytona/workspace/skills")
```

### Anti-Patterns to Avoid
- **Hardcoding Workspace ID:** Always use `MINERVA_WORKSPACE_ID` for resolution in OSS mode.
- **Sharing User Table:** Never mix `users` (developers) with `external_identities` (end-users) to prevent privilege escalation.

|---------|-------------|-------------|-----|
| Identity Hashing | Custom hashing | Raw passthrough | Context specifies raw ID passthrough to let developer control format. |
| DB Migrations | Raw SQL | Alembic | Ensure versioned, reversible schema changes. |

## Common Pitfalls

### Pitfall 1: Symlink Verification Timing
**What goes wrong:** Verification check runs before `ln -s` finishes.
**Why it happens:** `sandbox.process.exec` is async but might return before filesystem sync.
**How to avoid:** Use polling in the verification check with a short timeout.

### Pitfall 2: Orphaned Sandboxes
**What goes wrong:** Old sandboxes keep running after mount layout change.
**Why it happens:** Upgrade doesn't explicitly stop old instances.
**How to avoid:** Teardown all active sandboxes as part of the migration/upgrade procedure.

## Code Examples

### Symlink Creation in Daytona
```python
# Create workspace directory first
await sandbox.fs.create_folder("/home/daytona/workspace", "700")

# Symlink required identity files
identity_files = ["AGENT.md", "SOUL.md", "IDENTITY.md"]
for f in identity_files:
    await sandbox.process.exec(f"ln -s /workspace/pack/{f} /home/daytona/workspace/{f}")

# Symlink skills directory
await sandbox.process.exec("ln -s /workspace/pack/skills /home/daytona/workspace/skills")
```

### External Identity Model
```python
class ExternalIdentity(Base):
    __tablename__ = "external_identities"
    workspace_id = Column(UUID(as_uuid=True), ForeignKey("workspaces.id"), primary_key=True)
    external_user_id = Column(String(255), primary_key=True)
    created_at = Column(DateTime, default=datetime.utcnow, nullable=False)
```
