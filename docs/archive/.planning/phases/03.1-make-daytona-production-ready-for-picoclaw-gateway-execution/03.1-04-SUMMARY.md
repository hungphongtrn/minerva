# Phase 03.1 Plan 04: Daytona Base Image Guardrails Summary

## One-Liner

Deterministic base-image preparation guardrails with strict mode validation, AsyncDaytona preflight CLI, and progressive-disclosure operator runbook.

## What Was Built

### 1. Base Image Contract Validation (src/config/settings.py, src/infrastructure/sandbox/providers/daytona.py)

**New Settings:**
- `DAYTONA_BASE_IMAGE_STRICT_MODE` - Enforces digest-pinned images in production
- `DAYTONA_BASE_IMAGE_DIGEST_REQUIRED` - Require digest format (implied by strict mode)

**New Error Type:**
- `SandboxImageContractError` - Typed error for contract violations with remediation guidance

**Validation Logic:**
- `_validate_base_image_contract()` - Fail-fast validation on provider initialization
- Validates image uses digest format: `repo/image@sha256:...`
- Rejects mutable tags like `:latest`, `:v1`
- Stamps contract metadata labels on all provisioned sandboxes

### 2. AsyncDaytona Preflight CLI (src/scripts/daytona_base_image_preflight.py)

**Features:**
- Provisions disposable sandbox from candidate image
- Validates identity files (AGENT.md, SOUL.md, IDENTITY.md, skills/)
- Checks gateway endpoint resolution
- Guaranteed cleanup via `finally` block
- Exit codes: 0 (pass), 1 (fail), 2 (config error), 3 (unexpected)
- JSON output for CI gates
- Verbose mode for interactive debugging

**Usage:**
```bash
# Basic validation
uv run python src/scripts/daytona_base_image_preflight.py \
    --image registry/picoclaw@sha256:abc123...

# JSON output for CI
uv run python src/scripts/daytona_base_image_preflight.py \
    --image registry/picoclaw@sha256:abc123... \
    --json

# Verbose with custom timeout
uv run python src/scripts/daytona_base_image_preflight.py \
    --image registry/picoclaw@sha256:abc123... \
    --verbose --timeout 300
```

### 3. Operator Runbook (src/infrastructure/sandbox/providers/daytona_base_image_prep.md)

**Structure:**
1. **Quick Start** - Single command validation
2. **Prerequisites** - Credentials and environment setup
3. **Base Image Contract** - Required files and services
4. **Build Workflow** - Dockerfile creation, building, tagging
5. **Preflight Validation** - All CLI options with examples
6. **Rollout Workflow** - CI/CD integration, production promotion
7. **Rollback Workflow** - Emergency procedures
8. **Troubleshooting** - Common failures and fixes
9. **Advanced Configuration** - Custom IDs, self-hosted Daytona
10. **Security Considerations** - Digest pinning, strict mode

## Key Decisions

1. **Strict mode as opt-in:** Rather than forcing strict mode always, made it a toggle so development can use mutable tags while production requires digests.

2. **Existing error type extension:** `SandboxImageContractError` extends `SandboxConfigurationError` to reuse existing error mapping infrastructure.

3. **Metadata labeling:** Stamped `picoclaw.base_image` and `picoclaw.base_image_strict` labels on all sandboxes for observability.

4. **CLI as standalone script:** Chose script over subcommand for simpler CI integration (just run Python file).

5. **Progressive disclosure runbook:** Quick Start at top for common case, detailed sections below for edge cases and troubleshooting.
