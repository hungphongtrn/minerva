## User Constraints (from CONTEXT.md)

### Locked Decisions

### Developer setup flow
- Single CLI bootstrap tool named `minerva`, distributed as a uv CLI tool (`uv tool install` or `uvx`)
- Agent-pack-first workflow: developer prepares their agent pack files (AGENT.md, SOUL.md, IDENTITY.md, skills/), then points the CLI at them
- Full preflight validation with checklist report: each dependency gets green/red status with actionable fix instructions (e.g. "DAYTONA_API_URL is set but host is unreachable at https://...")
- Blocks on critical dependency failures, proceeds with warnings for optional deps
- `minerva migrate` is separate from `minerva serve` — server checks schema version at startup and refuses to start if behind, but never auto-migrates (safe for multi-replica k8s)

### CLI command surface
- 6 focused commands, each does one thing:
  - `minerva init` — generate `.env.example`, validate configuration
  - `minerva migrate` — run database migrations (Alembic under the hood)
  - `minerva serve` — start the orchestrator API server
  - `minerva scaffold` — create a starter agent pack with minimal files and inline comments
  - `minerva register` — validate agent pack structure and register it with the orchestrator
  - `minerva snapshot build` — build Picoclaw base snapshot via Daytona Declarative Builder

### Configuration contract
- All config via environment variables only (12-factor, k8s/serverless native)
- CLI generates a `.env.example` with all vars documented, grouped by service
- Required dependencies: Postgres and Daytona (server refuses to start without them)
- Optional dependencies: S3/Minio (no checkpoints without it) and LLM key/endpoint (no agent execution without it) — server starts but logs warnings
- Specific diagnostic errors on preflight failure: reports exactly what's wrong and which env var to set

### Template pack authoring
- `minerva scaffold` generates minimal agent pack files with inline comments explaining each section (AGENT.md, SOUL.md, IDENTITY.md, skills/)
- `minerva register` validates pack structure before registering: checks required files exist, validates format, reports missing/malformed files with clear errors
- Developer iterates by editing files on disk and re-running `minerva register` — stale detection via source digest flags if pack changed since last register

### Deployment topology
- **Local dev:** Docker Compose full stack — orchestrator + Postgres + Minio. Daytona is external (developer runs their own instance or uses Daytona Cloud)
- **Production:** Single process (uvicorn/gunicorn) — developer builds from source (Dockerfile provided, no pre-built registry image). Developer brings all infra externally
- Schema check + separate migrate for production multi-replica deployments

### Daytona sandbox provisioning strategy
- **Base image:** Pre-built Daytona Snapshot using Picoclaw's own Docker image directly (no modifications). Built from source via Declarative Builder. Snapshot rebuilds only when Picoclaw version changes
- **Picoclaw source:** Cloned from a Git repo URL configured by the developer (env var). `minerva snapshot build` pulls the repo and builds via Declarative Builder
- **Agent pack files:** Mounted via Daytona Volume (one volume per agent pack). Orchestrator syncs files to the volume via Daytona SDK during `minerva register`
- **Runtime config:** config.json generated by the orchestrator and written via Daytona SDK file API post-provisioning (includes bridge auth token, channel config, LLM config)
- **Startup validation:** Server checks if the configured snapshot exists in Daytona before accepting requests. If missing, startup fails with clear error telling developer to run `minerva snapshot build`

### End-user API surface
- REST API: POST /runs to send messages, SSE streaming response with typed events
- Developer-managed auth passthrough: developer's gateway sets X-User-ID header after authenticating the end-user. Minerva trusts it as an opaque string identifier (no format validation, developer is responsible for uniqueness)
- Trust boundary is network-level isolation only — developer puts minerva behind their own API gateway. No additional auth layer in minerva itself
- Optional X-Session-ID header for multi-session support: if omitted, orchestrator generates one. Same user + same session = same conversation thread. Same user + different session = separate thread, same sandbox

### SSE event contract
- Implement Phase 5 event types from the start: message, tool_call, tool_result, ui_patch, state_update, error, plus lifecycle events (queued, running, completed, failed)
- Orchestrator transforms Picoclaw's bridge response into typed SSE events — Picoclaw's internal format is hidden from end-users
- Provisioning status events emitted during cold start (first request for a new user)

### User onboarding flow
- Auto-provision on first request: new X-User-ID triggers automatic workspace + sandbox creation (zero setup per end-user by the developer)
- Inline provisioning with SSE status: first request is slower (sandbox provisioning), SSE stream starts with provisioning events, then agent response once ready
- Picoclaw owns session state (conversation history, memory) — orchestrator passes session ID to Picoclaw via bridge, does not store messages

### Concurrent request handling
- Queue per user: sandbox processes one message at a time. Second request waits in queue until the first completes, then executes

### Pack updates in production
- Drain old, new gets update: existing sandboxes keep running with old pack version, new sandboxes get new version
- Digest-based stale detection (reuse existing Phase 2 source_digest): at routing time, if registered pack digest differs from sandbox's pack digest, mark sandbox for drain and provision new one on next request
- Stale sandboxes drain via idle TTL (no aggressive teardown)
- Log stale detection events with old/new digest, workspace ID, and sandbox ID for rollout tracking

### Checkpoint behavior
- No S3 = ephemeral sessions: sandboxes lose session state when they stop (idle TTL, pack updates, crashes). Developers who want persistent sessions must configure S3. Document this tradeoff clearly

### LLM configuration passthrough
- LLM credentials (API key + endpoint) are part of the generated config.json written to each sandbox via Daytona SDK
- Orchestrator injects them from the developer's env vars into the Picoclaw config during sandbox provisioning
- Single LLM provider config (one API key + endpoint per deployment)

### End-user error experience
- Typed SSE error events with category (provisioning_failed, agent_error, rate_limited) and human-readable message. No internal details leaked
- Sanitized errors to end-user, full details (sandbox IDs, bridge failures, stack traces) in structured server logs
- One auto-retry for transient failures (Daytona unavailable, network blip), then emit error event and close stream. End-user can retry by sending new request

### Observability for operators
- /health and /ready endpoints: health returns component status (DB, Daytona, S3), ready returns whether server can accept traffic (k8s-native)
- Structured JSON logs (one JSON object per line, compatible with ELK/Datadog/CloudWatch)
- Prometheus /metrics endpoint with request counts, latencies, sandbox counts, error rates
- Sandbox lifecycle logging: create, ready, idle-stop, error events with workspace/user context for end-to-end traceability

### Sandbox resource limits
- Use Daytona's default resource allocation. Developer does not configure CPU/memory per sandbox. Daytona manages resource boundaries

### OpenCode's Discretion
- Exact SSE event envelope schema (field names, nesting)
- Internal queue implementation for per-user request ordering
- Preflight validation timing and parallelism
- `.env.example` grouping and documentation style
- Docker Compose service naming and networking
- Log correlation ID format
- Prometheus metric naming conventions
- Exact scaffold file content and comment wording

### Deferred Ideas (OUT OF SCOPE)

- Admin/management HTTP API (list sandboxes, force-stop users, view status) — deferred to business version
- Pre-warmed sandbox pool for instant first-request response — future optimization phase
- Multi-provider LLM configuration with model mapping — future enhancement
- Pre-built Docker image on public registry (ghcr.io) — currently build from source only
- Developer admin dashboard UI — business version

## Summary

Phase 03.2 is primarily a packaging and “real wiring” phase: turn the existing Minerva orchestrator (FastAPI + SQLAlchemy + Alembic + existing lifecycle/persistence/bridge services) into an OSS-deployable product with a developer-facing `minerva` CLI and an end-user-facing `/runs` SSE streaming API. The biggest technical shifts versus earlier phases are: (1) move from API-key auth to network-trusted `X-User-ID` passthrough for the end-user API, (2) add SSE typed event streaming (including provisioning status events), and (3) make Daytona integration real for agent packs using **Volumes** + **Sandbox file API** and for Picoclaw base runtime using **Snapshots** built from a repo via Daytona’s declarative `Image` builder.

The repo already has the right internal building blocks: deterministic agent pack digest (`AgentPack.source_digest`), pack validation/scaffold services, workspace lifecycle + lease locking, production-grade bridge service with token rotation, and sandbox orchestrator with bounded reprovision. The planning work should focus on: (a) defining the OSS API surface (new `/runs` + SSE contract) without breaking existing `/api/v1/*` APIs, (b) adding a proper uv-installed CLI entry point (requires adding a build system + `[project.scripts]`), (c) implementing volume-per-pack-(digest) semantics to satisfy “drain old, new gets update”, and (d) adding operator-grade `/ready` + `/metrics` + JSON logs.

**Primary recommendation:** Use Daytona Async SDK primitives directly (SnapshotService + VolumeService + Sandbox.fs + Sandbox.get_preview_link) and standardize on FastAPI SSE via `EventSourceResponse` (FastAPI 0.135+) plus Prometheus via `prometheus-fastapi-instrumentator`.

### Pattern 1: SSE Streaming With Typed Events (FastAPI 0.135+)
**What:** Implement `/runs` as a POST that returns an SSE stream of typed events (queued/running/completed/failed + domain events). Use `ServerSentEvent` for explicit event typing.

**When to use:** Every end-user run request (Context decision: “POST /runs … SSE streaming response”).

**Example:**
```python
from collections.abc import AsyncIterable

from fastapi import APIRouter
from fastapi.sse import EventSourceResponse, ServerSentEvent

router = APIRouter()


@router.post("/runs", response_class=EventSourceResponse)
async def runs() -> AsyncIterable[ServerSentEvent]:
    # queued / provisioning events first
    yield ServerSentEvent(event="queued", data={"run_id": "..."})
    yield ServerSentEvent(event="provisioning", data={"step": "sandbox_create"})

    # agent events
    yield ServerSentEvent(event="message", data={"role": "assistant", "content": "..."})

    # lifecycle terminal
    yield ServerSentEvent(event="completed", data={"run_id": "..."})
```
Source: https://fastapi.tiangolo.com/tutorial/server-sent-events/

### Pattern 2: Daytona Volumes + File API For Agent Pack Sync
**What:** Treat agent pack artifacts as static identity files (AGENTS.md invariants). Registering a pack materializes content into a Daytona Volume, and provisioning mounts that Volume into the Sandbox. Use `sandbox.fs.upload_files()` for sync.

**When to use:** `minerva register` (sync pack -> volume) and sandbox provisioning (mount volume).

**Key SDK primitives:**
- `await daytona.volume.get(name, create=True)` (create/get volume)
- `VolumeMount(volume_id=..., mount_path="/workspace/pack")` (mount)
- `await sandbox.fs.upload_files([...])` (upload pack files)

**Example:**
```python
from daytona import AsyncDaytona
from daytona import CreateSandboxFromSnapshotParams
from daytona import VolumeMount
from daytona import FileUpload


async with AsyncDaytona() as daytona:
    volume = await daytona.volume.get("agent-pack-<digest>", create=True)

    params = CreateSandboxFromSnapshotParams(
        snapshot="picoclaw-base",
        volumes=[VolumeMount(volume_id=volume.id, mount_path="/workspace/pack")],
    )
    sandbox = await daytona.create(params)

    await sandbox.fs.upload_files(
        [
            FileUpload(source=b"...", destination="/workspace/pack/AGENT.md"),
            FileUpload(source=b"...", destination="/workspace/pack/SOUL.md"),
        ]
    )
```
Sources:
- Volumes + VolumeMount: https://www.daytona.io/docs/en/python-sdk/async/async-volume/
- Sandbox file API: https://www.daytona.io/docs/en/python-sdk/async/async-file-system/
- Sandbox creation params (volumes included): https://www.daytona.io/docs/en/python-sdk/async/async-daytona/

### Pattern 3: Snapshot Build With Daytona Declarative `Image`
**What:** Use `daytona.snapshot.create(CreateSnapshotParams(name=..., image=Image...))` where `Image` is built from a developer-provided Picoclaw repo checkout using `.add_local_dir()` + `.run_commands()` and optionally `.pip_install_from_pyproject()`.

**When to use:** `minerva snapshot build` and startup preflight “snapshot exists”.

**Example (skeleton):**
```python
from daytona import AsyncDaytona
from daytona import Image
from daytona import CreateSnapshotParams


image = (
    Image.base("python:3.12-slim")
    .add_local_dir("./picoclaw", "/opt/picoclaw")
    .workdir("/opt/picoclaw")
    .run_commands(
        "apt-get update && apt-get install -y git ca-certificates",
        "make build",  # whatever Picoclaw build expects
    )
)

async with AsyncDaytona() as daytona:
    await daytona.snapshot.create(
        CreateSnapshotParams(name="picoclaw-base", image=image),
        on_logs=lambda chunk: print(chunk, end=""),
    )
```
Source (Image builder API): https://www.daytona.io/docs/en/python-sdk/common/image/
Source (Snapshot service): https://www.daytona.io/docs/en/python-sdk/async/async-snapshot/

### Anti-Patterns to Avoid
- **Mutating a shared volume in place for pack updates:** breaks “drain old, new gets update”. Prefer volume-per-digest (immutable) and mount the digest-selected volume.
- **In-process per-user queue without documenting replica limitations:** multi-replica deployments will break ordering unless requests are sticky-routed or queue is external (Phase 4 territory).
- **Using FastAPI SSE APIs on older FastAPI versions:** `fastapi.sse` is documented as added in FastAPI 0.135.0; upgrading or using an SSE library is required.
- **Relying on synthetic gateway URL construction:** prefer Daytona preview link APIs (see AsyncSandbox `get_preview_link`) over string munging.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SSE protocol formatting/keepalive headers | manual `StreamingResponse` text formatting | FastAPI `EventSourceResponse` + `ServerSentEvent` (0.135+) | Handles SSE headers + ping keepalive per docs; less proxy breakage (Source: https://fastapi.tiangolo.com/tutorial/server-sent-events/) |
| Prometheus HTTP metrics plumbing | custom metrics middleware from scratch | `prometheus-fastapi-instrumentator` | Standard counters/histograms with low-cardinality defaults (Source: https://pypi.org/project/prometheus-fastapi-instrumentator/) |
| Daytona image build workflow | custom Docker build pipeline | Daytona `Image` + `SnapshotService.create` | Declarative builder supports `run_commands`, `add_local_dir`, `pip_install_from_pyproject` (Source: https://www.daytona.io/docs/en/python-sdk/common/image/ ) |
| Volume management | ad-hoc pack copying into sandbox filesystem | Daytona Volumes + Sandbox file API | Shared storage and explicit mounts; matches decisions (Sources: async volume + async filesystem docs) |
| CLI installation & entrypoints | “python -m …” docs only | uv `tool install` + `[project.scripts]` | Standard for uv-distributed CLIs; requires build-system (Source: https://docs.astral.sh/uv/concepts/projects/config/#entry-points ) |

**Key insight:** The OSS MVP is mostly integration glue. The highest leverage is using “blessed” primitives (SSE response class, Daytona Volume/Snapshot/FileSystem services, Prometheus instrumentation) to avoid subtle protocol/ops bugs.

## Common Pitfalls

### Pitfall 1: uv CLI entrypoints silently not working
**What goes wrong:** `uv run minerva` / `uv tool install` doesn’t expose the command even after adding `[project.scripts]`.

**Why it happens:** uv requires a build system to be defined to build/install the project for entry points.

**How to avoid:** Add a `[build-system]` to `pyproject.toml` and configure which package(s) are included.

**Warning signs:** “Failed to spawn … No such file or directory” when running a script entry point.

Source: https://docs.astral.sh/uv/concepts/projects/config/#entry-points

### Pitfall 2: Pack update semantics conflict with “one volume per agent pack”
**What goes wrong:** Updating the agent pack updates the Volume content; old sandboxes see new content, so “drain old, new gets update” is violated.

**Why it happens:** Daytona Volumes are shared storage; if multiple sandboxes mount the same volume, they see the same filesystem.

**How to avoid:** Name volumes by digest (`agent-pack-{pack_id}-{digest}`), treat them immutable, and mount the digest-selected volume.

**Warning signs:** Old sandbox’s `pack_digest` != mounted volume content digest.

Source: https://www.daytona.io/docs/en/python-sdk/async/async-volume/

### Pitfall 3: SSE buffering and “no events until completion”
**What goes wrong:** Client receives events in a batch at the end (or times out), especially behind proxies.

**Why it happens:** Missing SSE headers/keepalive; proxy buffering.

**How to avoid:** Prefer `EventSourceResponse` (FastAPI SSE) which sets `Cache-Control: no-cache` and `X-Accel-Buffering: no` and emits ping comments.

Source: https://fastapi.tiangolo.com/tutorial/server-sent-events/

### Pitfall 4: Prometheus metrics with gunicorn multi-workers
**What goes wrong:** Duplicate time series / partial metrics when using multiple processes.

**Why it happens:** `prometheus_client` requires multiprocess mode for multiple worker processes.

**How to avoid:** Either run single worker for MVP, or set up multiprocess mode (PROMETHEUS_MULTIPROC_DIR + registry collector) as documented.

Source: https://prometheus.github.io/client_python/exporting/http/fastapi-gunicorn/

## Open Questions

1. **How should `X-User-ID` map onto the existing identity schema?**
   - What we know: existing tables expect `users.email` non-null; OSS flow wants opaque string user IDs.
   - What's unclear: whether to add `external_user_id` to `users`, create a new mapping table, or bypass `users` entirely and key workspaces directly.
   - Recommendation: introduce an `external_identities` mapping table (`external_user_id` unique -> `workspace_id`) to avoid forcing email semantics.

2. **What is the canonical “volume per agent pack” definition?**
   - What we know: decisions require “one volume per agent pack” and “drain old/new gets update”.
   - What's unclear: whether volumes are mutable or immutable across updates.
   - Recommendation: interpret “agent pack” as “agent pack revision (digest)” and name volumes by digest.

3. **SSE event envelope schema details**
   - What we know: required event types (message/tool_call/tool_result/ui_patch/state_update/error + queued/running/completed/failed).
   - What's unclear: exact fields, `id` semantics, and whether to support `Last-Event-ID` resumption now.
   - Recommendation: include `id`, `type`, `ts`, `run_id`, and a `data` payload; implement `Last-Event-ID` only if transcript retrieval is added later.
