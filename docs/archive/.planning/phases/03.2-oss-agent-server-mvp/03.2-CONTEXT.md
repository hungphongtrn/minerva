# Phase 3.2: OSS Agent Server MVP - Context

**Gathered:** 2026-03-02
**Status:** Ready for planning

<domain>
## Phase Boundary

End-to-end OSS flow: template agent pack to deployed multi-tenant agent server with sandbox routing, ready for developers who bring their own infrastructure (Postgres, S3/Minio, Daytona, LLM key/endpoints). Each deployed server is a specific ready agent handling multiple users in respective sandboxes. This phase delivers the developer-facing CLI, runtime server, end-user API surface, and Daytona integration to make the existing orchestrator usable as a standalone OSS product.

This phase does NOT include: business platform UI, admin management API, multi-provider LLM routing, pre-warmed sandbox pools, or any SaaS-specific features. Those belong in the business version or future phases.

</domain>

<decisions>
## Implementation Decisions

### Developer setup flow
- Single CLI bootstrap tool named `minerva`, distributed as a uv CLI tool (`uv tool install` or `uvx`)
- Agent-pack-first workflow: developer prepares their agent pack files (AGENT.md, SOUL.md, IDENTITY.md, skills/), then points the CLI at them
- Full preflight validation with checklist report: each dependency gets green/red status with actionable fix instructions (e.g. "DAYTONA_API_URL is set but host is unreachable at https://...")
- Blocks on critical dependency failures, proceeds with warnings for optional deps
- `minerva migrate` is separate from `minerva serve` — server checks schema version at startup and refuses to start if behind, but never auto-migrates (safe for multi-replica k8s)

### CLI command surface
- 6 focused commands, each does one thing:
  - `minerva init` — generate `.env.example`, validate configuration
  - `minerva migrate` — run database migrations (Alembic under the hood)
  - `minerva serve` — start the orchestrator API server
  - `minerva scaffold` — create a starter agent pack with minimal files and inline comments
  - `minerva register` — validate agent pack structure and register it with the orchestrator
  - `minerva snapshot build` — build Picoclaw base snapshot via Daytona Declarative Builder

### Configuration contract
- All config via environment variables only (12-factor, k8s/serverless native)
- CLI generates a `.env.example` with all vars documented, grouped by service
- Required dependencies: Postgres and Daytona (server refuses to start without them)
- Optional dependencies: S3/Minio (no checkpoints without it) and LLM key/endpoint (no agent execution without it) — server starts but logs warnings
- Specific diagnostic errors on preflight failure: reports exactly what's wrong and which env var to set

### Template pack authoring
- `minerva scaffold` generates minimal agent pack files with inline comments explaining each section (AGENT.md, SOUL.md, IDENTITY.md, skills/)
- `minerva register` validates pack structure before registering: checks required files exist, validates format, reports missing/malformed files with clear errors
- Developer iterates by editing files on disk and re-running `minerva register` — stale detection via source digest flags if pack changed since last register

### Deployment topology
- **Local dev:** Docker Compose full stack — orchestrator + Postgres + Minio. Daytona is external (developer runs their own instance or uses Daytona Cloud)
- **Production:** Single process (uvicorn/gunicorn) — developer builds from source (Dockerfile provided, no pre-built registry image). Developer brings all infra externally
- Schema check + separate migrate for production multi-replica deployments

### Daytona sandbox provisioning strategy
- **Base image:** Pre-built Daytona Snapshot using Picoclaw's own Docker image directly (no modifications). Built from source via Declarative Builder. Snapshot rebuilds only when Picoclaw version changes
- **Picoclaw source:** Cloned from a Git repo URL configured by the developer (env var). `minerva snapshot build` pulls the repo and builds via Declarative Builder
- **Agent pack files:** Mounted via Daytona Volume (one volume per agent pack). Orchestrator syncs files to the volume via Daytona SDK during `minerva register`
- **Runtime config:** config.json generated by the orchestrator and written via Daytona SDK file API post-provisioning (includes bridge auth token, channel config, LLM config)
- **Startup validation:** Server checks if the configured snapshot exists in Daytona before accepting requests. If missing, startup fails with clear error telling developer to run `minerva snapshot build`

### End-user API surface
- REST API: POST /runs to send messages, SSE streaming response with typed events
- Developer-managed auth passthrough: developer's gateway sets X-User-ID header after authenticating the end-user. Minerva trusts it as an opaque string identifier (no format validation, developer is responsible for uniqueness)
- Trust boundary is network-level isolation only — developer puts minerva behind their own API gateway. No additional auth layer in minerva itself
- Optional X-Session-ID header for multi-session support: if omitted, orchestrator generates one. Same user + same session = same conversation thread. Same user + different session = separate thread, same sandbox

### SSE event contract
- Implement Phase 5 event types from the start: message, tool_call, tool_result, ui_patch, state_update, error, plus lifecycle events (queued, running, completed, failed)
- Orchestrator transforms Picoclaw's bridge response into typed SSE events — Picoclaw's internal format is hidden from end-users
- Provisioning status events emitted during cold start (first request for a new user)

### User onboarding flow
- Auto-provision on first request: new X-User-ID triggers automatic workspace + sandbox creation (zero setup per end-user by the developer)
- Inline provisioning with SSE status: first request is slower (sandbox provisioning), SSE stream starts with provisioning events, then agent response once ready
- Picoclaw owns session state (conversation history, memory) — orchestrator passes session ID to Picoclaw via bridge, does not store messages

### Concurrent request handling
- Queue per user: sandbox processes one message at a time. Second request waits in queue until the first completes, then executes

### Pack updates in production
- Drain old, new gets update: existing sandboxes keep running with old pack version, new sandboxes get new version
- Digest-based stale detection (reuse existing Phase 2 source_digest): at routing time, if registered pack digest differs from sandbox's pack digest, mark sandbox for drain and provision new one on next request
- Stale sandboxes drain via idle TTL (no aggressive teardown)
- Log stale detection events with old/new digest, workspace ID, and sandbox ID for rollout tracking

### Checkpoint behavior
- No S3 = ephemeral sessions: sandboxes lose session state when they stop (idle TTL, pack updates, crashes). Developers who want persistent sessions must configure S3. Document this tradeoff clearly

### LLM configuration passthrough
- LLM credentials (API key + endpoint) are part of the generated config.json written to each sandbox via Daytona SDK
- Orchestrator injects them from the developer's env vars into the Picoclaw config during sandbox provisioning
- Single LLM provider config (one API key + endpoint per deployment)

### End-user error experience
- Typed SSE error events with category (provisioning_failed, agent_error, rate_limited) and human-readable message. No internal details leaked
- Sanitized errors to end-user, full details (sandbox IDs, bridge failures, stack traces) in structured server logs
- One auto-retry for transient failures (Daytona unavailable, network blip), then emit error event and close stream. End-user can retry by sending new request

### Observability for operators
- /health and /ready endpoints: health returns component status (DB, Daytona, S3), ready returns whether server can accept traffic (k8s-native)
- Structured JSON logs (one JSON object per line, compatible with ELK/Datadog/CloudWatch)
- Prometheus /metrics endpoint with request counts, latencies, sandbox counts, error rates
- Sandbox lifecycle logging: create, ready, idle-stop, error events with workspace/user context for end-to-end traceability

### Sandbox resource limits
- Use Daytona's default resource allocation. Developer does not configure CPU/memory per sandbox. Daytona manages resource boundaries

### OpenCode's Discretion
- Exact SSE event envelope schema (field names, nesting)
- Internal queue implementation for per-user request ordering
- Preflight validation timing and parallelism
- `.env.example` grouping and documentation style
- Docker Compose service naming and networking
- Log correlation ID format
- Prometheus metric naming conventions
- Exact scaffold file content and comment wording

</decisions>

<specifics>
## Specific Ideas

- The orchestrator is minerva, not Picoclaw. Picoclaw only lives inside sandboxes. The CLI and server are the orchestrator.
- Picoclaw is a Go binary built from source — the Dockerfile is at `reference_repos/picoclaw/Dockerfile`. Gateway listens on port 18790 with `/health` and `/bridge/execute` endpoints.
- The bridge service (`src/services/picoclaw_bridge_service.py`) already implements health-first fail-closed bridge communication with typed errors and token rotation. This should be extended, not rewritten.
- Existing simulated implementations in `src/infrastructure/sandbox/providers/daytona.py` (identity file verification, pack materialization) need to be replaced with real Daytona SDK operations using Volumes and file API.
- Agent pack files are static identity files per AGENTS.md invariants: "Static identity files are mounted during sandbox creation: AGENT.md, SOUL.md, IDENTITY.md, skills/". Checkpoint data is memory/session state only.
- Daytona Volumes are FUSE-based — slower than local filesystem but support shared access across sandboxes. Acceptable for agent pack files (read-heavy, small files).
- Daytona Declarative Builder supports `run_commands`, `add_local_dir`, `pip_install_from_pyproject` — sufficient to build Picoclaw from source and register as a pre-built snapshot.
- For the business version later: admin management API, multi-provider LLM, platform UI for creating agents.

</specifics>
