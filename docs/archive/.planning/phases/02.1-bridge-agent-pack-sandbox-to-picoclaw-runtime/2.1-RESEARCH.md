# Phase 2.1: Bridge Agent Pack Sandbox to Picoclaw Runtime - Research

**Researched:** 2026-02-25
**Domain:** Picoclaw Gateway Runtime Integration / Sandbox Bridge Architecture
**Confidence:** HIGH

## Summary

This phase bridges Phase 2's workspace lifecycle infrastructure with Picoclaw runtime execution capabilities. The goal is to enable Minerva to invoke Picoclaw runtime inside provisioned sandboxes and return execution results through the existing `/runs` API.

**Key architectural decision:** Use Picoclaw `gateway` mode (long-running HTTP daemon) instead of `agent` mode (one-shot CLI). This provides:
- Continuous readiness via `/health` endpoint
- Message bus architecture for bidirectional communication
- Session persistence capabilities
- Standard HTTP interface for bridge integration

**Primary recommendation:** Build an HTTP bridge service that:
1. Polls sandbox `/health` before routing requests
2. Authenticates via bearer token (injected via env vars)
3. Transforms Minerva envelope to Picoclaw InboundMessage format
4. Returns synchronous responses (Phase 2.1 scope)
5. Maintains session continuity across respawns via workspace+pack scoped session keys

### Pattern 1: Health-Aware Routing

**What:** Poll `/health` before every execution request; fail-closed on unhealthy.

**When to use:** Always. This is the Phase 2 control-plane behavior style.

**Example:**
```python
# Source: Phase 2 orchestrator pattern
async def execute_via_bridge(sandbox_info: SandboxInfo, request: RunRequest):
    # 1. Health check with timeout
    health = await poll_health(
        sandbox_url,
        timeout=10,
        retries=3,
        backoff=1.0
    )
    if not health or health.status != "ok":
        raise SandboxUnavailableError("Health check failed")
    
    # 2. Execute only if healthy
    return await bridge.execute(request)
```

### Pattern 2: Session Scoping

**What:** Session key format encodes workspace + agent_pack to prevent cross-contamination.

**When to use:** All execution requests requiring session continuity.

**Format:** `minerva:{workspace_id}:{agent_pack_id}:{run_id}`

**Example:**
```python
# Source: Picoclaw session docs
session_key = f"minerva:{workspace_id}:{agent_pack_id}:{run_id}"

# Picoclaw stores in: workspace/sessions/{session_key}.json
# This ensures:
# - Different packs in same workspace = different sessions
# - Same pack after respawn = same session (if session dir persisted)
```

### Pattern 3: Config.json Generation

**What:** Generate Picoclaw config per-sandbox with bridge-only channel.

**When to use:** During sandbox provisioning or pack binding.

**Structure:**
```json
{
  "agents": {
    "defaults": {

      "restrict_to_workspace": true,

  },
  "model_list": [
    {
      "model_name": "primary",
      "model": "{{LLM_MODEL}}",
      "api_key": "{{LLM_API_KEY}}"
    }
  ],
  "channels": {
    "bridge": {
      "enabled": true,
      "auth_token": "{{BRIDGE_AUTH_TOKEN}}"
    },
    "telegram": { "enabled": false },
    "discord": { "enabled": false }

```

**Key principles:**
- Bridge channel enabled, all public channels disabled
- Credentials from environment variables (not persisted)
- Workspace points to mounted pack root
- Generated deterministically from workspace+pack context

### Pattern 4: Pack Materialization

**What:** Snapshot copy/sync pack content into sandbox workspace.

**When to use:** During sandbox provisioning with agent pack binding.

**Provider-specific:**
- **Local Compose:** Volume mount pack directory
- **Daytona:** Copy pack content into workspace filesystem

**Verification:** Pack digest stored in sandbox metadata; fail-closed on mismatch.

### Anti-Patterns to Avoid

- **Live host bind:** Don't mount host pack directory directly; use snapshot copy for deterministic behavior
- **Shared config:** Don't reuse config.json across workspaces; each sandbox gets isolated config
- **Synchronous health polling without timeout:** Always cap health checks to prevent indefinite blocking
- **Session key without scope prefix:** Always prefix with `minerva:` to avoid collisions with native Picoclaw sessions

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Session persistence | Custom session store | Picoclaw native session files | Picoclaw already handles session serialization, file locking, and cleanup |
| Health endpoint | Custom health check | Picoclaw `/health` and `/ready` | Built-in, returns structured status with uptime and checks |
| Message protocol | Raw socket protocol | Picoclaw message bus abstractions | InboundMessage/OutboundMessage provide consistent interface |
| Config templating | String concatenation | Jinja2 or structured JSON generation | Prevents injection, handles escaping, supports validation |
| Auth token generation | `random.random()` | `secrets.token_urlsafe(32)` | Cryptographically secure, URL-safe |
| Sandbox discovery | Scan network | Provider's get_active_sandbox() | Provider already tracks workspace→sandbox mapping |

**Key insight:** Picoclaw is designed to be embedded. Leverage its native capabilities (sessions, health checks, config loading) rather than building parallel infrastructure.

## Common Pitfalls

### Pitfall 1: Session Loss on Respawn

**What goes wrong:** Sandbox stops due to idle TTL, new sandbox provisioned, but session history is lost.

**Why it happens:** Default Picoclaw sessions are filesystem-local to the sandbox.

**How to avoid:** 
- Mount session directory to persistent volume (provider-specific)
- Or implement session backup/restore in bridge
- Or accept session loss as Phase 2.1 limitation (document in API)

**Warning signs:** "Agent doesn't remember previous context" after idle period.

### Pitfall 2: Config.json Secret Leakage

**What goes wrong:** API keys written to config.json in sandbox filesystem.

**Why it happens:** Picoclaw config format requires API keys in model_list.

**How to avoid:**
- Pass sensitive values via environment variables
- Use template placeholders in config.json: `"api_key": "{{ENV:LLM_API_KEY}}"`
- Configure Picoclaw to read from env (see config.example.json patterns)

**Warning signs:** API keys visible in sandbox filesystem inspection.

### Pitfall 3: Health Check False Positives

**What goes wrong:** Health check passes but execution fails immediately.

**Why it happens:** Health checks `/health` and `/ready` verify process is running, not that model provider is connected.

**How to avoid:**
- Check provider connectivity in `/health` response (if available)
- Retry execution with backoff on initial failures
- Fail-closed: treat execution failure as unhealthy sandbox

**Warning signs:** Intermittent "sandbox unavailable" errors despite healthy status.

### Pitfall 4: Pack Stale Detection Race

**What goes wrong:** Pack source changes after sandbox provisioning, but sandbox uses old version.

**Why it happens:** Snapshot copy happens at provision time; source is not continuously synced.

**How to avoid:**
- Store pack digest in sandbox metadata
- Check digest on execution; fail-closed if mismatch
- Require re-provision on stale detection

**Warning signs:** Agent behavior doesn't match updated pack source.

### Pitfall 5: Cross-Workspace Session Bleed

**What goes wrong:** Session from workspace A accessible in workspace B.

**Why it happens:** Session key collision or improper scoping.

**How to avoid:**
- Strict session key format: `minerva:{workspace_id}:{pack_id}:{run_id}`
- Validate workspace isolation before execution
- Reject cross-workspace session keys fail-closed

**Warning signs:** User sees conversation history from different workspace.

## Code Examples

### Bridge Service Interface

```python
# Source: Pattern based on Phase 2 services
class PicoclawBridgeService:
    """Bridge between Minerva orchestrator and Picoclaw runtime."""
    
    DEFAULT_HEALTH_TIMEOUT = 10
    DEFAULT_HEALTH_RETRIES = 3
    DEFAULT_BRIDGE_TIMEOUT = 300  # 5 minutes for synchronous execution
    
    def __init__(
        self,
        health_timeout: int = DEFAULT_HEALTH_TIMEOUT,
        bridge_timeout: int = DEFAULT_BRIDGE_TIMEOUT,
    ):
        self.health_timeout = health_timeout
        self.bridge_timeout = bridge_timeout
    
    async def execute(
        self,
        sandbox: SandboxInstance,
        request: MinervaRunRequest,
        principal: Any,
    ) -> BridgeExecutionResult:
        """Execute request via bridge with health check and auth."""
        
        # 1. Resolve sandbox URL from provider info
        sandbox_url = self._resolve_sandbox_url(sandbox)
        
        # 2. Health check with retries
        health = await self._poll_health(sandbox_url)
        if not health or health.status != "ok":
            raise SandboxUnavailableError(
                f"Sandbox {sandbox.id} health check failed"
            )
        
        # 3. Transform Minerva envelope to Picoclaw format
        bridge_request = self._transform_request(request, sandbox, principal)
        
        # 4. Execute via HTTP bridge
        try:
            response = await self._http_post(
                f"{sandbox_url}/bridge/execute",
                headers={
                    "Authorization": f"Bearer {self._get_auth_token(sandbox)}",
                    "Content-Type": "application/json",
                },
                json=bridge_request,
                timeout=self.bridge_timeout,
            )
            
            # 5. Transform response back to Minerva format
            return self._transform_response(response)
            
        except TimeoutError:
            raise BridgeTimeoutError(
                f"Execution exceeded {self.bridge_timeout}s timeout"
            )
    
    def _transform_request(
        self,
        request: MinervaRunRequest,
        sandbox: SandboxInstance,
        principal: Any,
    ) -> dict:
        """Transform Minerva envelope to Picoclaw InboundMessage format."""
        return {
            "channel": "bridge",
            "sender_id": str(getattr(principal, "id", "guest")),
            "chat_id": request.run_id,
            "content": request.input.get("message", ""),
            "session_key": self._generate_session_key(sandbox, request),
            "metadata": {
                "workspace_id": str(sandbox.workspace_id),
                "agent_pack_id": str(sandbox.agent_pack_id) if sandbox.agent_pack_id else None,
                "run_id": request.run_id,
            }
        }
    
    def _generate_session_key(
        self,
        sandbox: SandboxInstance,
        request: MinervaRunRequest,
    ) -> str:
        """Generate scoped session key for continuity."""
        pack_id = sandbox.agent_pack_id or "default"
        return f"minerva:{sandbox.workspace_id}:{pack_id}:{request.run_id}"
```

```python
async def mount_agent_pack(
    self,
    sandbox_ref: SandboxRef,
    pack_source_path: str,
) -> None:
    """Mount agent pack into local compose sandbox."""
    
    # In docker-compose.yml:
    # volumes:
    #   - {pack_source_path}:/workspace/pack:ro
    
    # Update sandbox metadata
    self._sandboxes[sandbox_ref.provider_ref]["pack_bound"] = True
    self._sandboxes[sandbox_ref.provider_ref]["pack_source_path"] = pack_source_path
```

### Provider Pack Materialization (Daytona)

```python
# Source: Pattern from daytona.py extension
async def materialize_agent_pack(
    self,
    sandbox_ref: SandboxRef,
    pack_source_path: str,
) -> None:
    """Copy agent pack into Daytona workspace filesystem."""
    
    # 1. Copy pack content into sandbox
    # Use Daytona SDK file operations
    async with AsyncDaytona(config=self._create_config()) as daytona:
        sandbox = await daytona.get(sandbox_ref.provider_ref)
        
        # Copy files into /workspace/pack
        await self._copy_pack_to_workspace(
            sandbox,
            source_path=pack_source_path,
            dest_path="/workspace/pack",
        )
        
        # 2. Store pack digest for staleness detection
        digest = await self._compute_pack_digest(pack_source_path)
        await self._set_sandbox_metadata(
            sandbox,
            {
                "pack_bound": True,
                "pack_source_path": pack_source_path,
                "pack_digest": digest,
            }
        )
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| One-shot agent execution | Gateway daemon mode | Phase 2.1 decision | Enables session persistence and health checking |
| Raw Picoclaw payloads | Minerva envelope | Phase 2.1 decision | Provides versioning, error typing, abstraction |
| Host bind mounts | Snapshot copy/sync | Phase 2.1 decision | Deterministic, portable across providers |
| Implicit session isolation | Explicit workspace+pack scoping | Phase 2.1 decision | Prevents cross-contamination |
| External channel configs | Bridge-only mode | Phase 2.1 decision | Simplifies security, defers public channel work |

**Deferred to later phases:**
- Token streaming and incremental events (deferred to streaming-focused phase)
- Async callback/job execution mode (deferred)
- Public chat channels (Telegram/Discord/etc.) in sandbox runtime (deferred)
- Session backup/restore for true continuity (Phase 2.1 accepts session loss on respawn as limitation)

## Open Questions

Things that couldn't be fully resolved:

1. **Picoclaw Gateway HTTP API surface**
   - What we know: Gateway exposes `/health` and `/ready`
   - What's unclear: Exact bridge execution endpoint (Picoclaw may need custom channel implementation)
   - Recommendation: Verify if custom bridge channel needed or use webhook-style endpoint

2. **Session persistence mechanism**
   - What we know: Picoclaw uses filesystem sessions in `workspace/sessions/`
   - What's unclear: How to persist sessions across sandbox respawns (volume mounts vs. backup/restore)
   - Recommendation: Accept session loss as Phase 2.1 limitation; document in API responses

3. **Bridge channel implementation**
   - What we know: Picoclaw supports custom channels via `Channel` interface
   - What's unclear: Whether HTTP bridge channel already exists or needs custom implementation
   - Recommendation: Research Picoclaw channel implementations; may need custom `BridgeChannel`

4. **Provider-specific pack materialization**
   - What we know: Local compose uses volume mounts, Daytona needs file copy
   - What's unclear: Exact Daytona SDK methods for bulk file copy
   - Recommendation: Implement provider-specific materialization strategies
