Purpose: Prevent AUTH_FAILED/401 cascades caused by DB/runtime token mismatch during provisioning and rotation.
Output: Consistent provision-time token injection + regression tests proving current+previous token lifecycle.
</objective>

<task type="auto">
  <name>task 1: Align provision-time gateway token with persisted token</name>
  <files>src/services/sandbox_orchestrator_service.py</files>
  <action>
Rewrite the provision-time token flow so the in-sandbox gateway config token and DB token are the same value.

Implementation requirements:
1) Stop generating an auth token inside `_generate_runtime_bridge_config()` without surfacing it.
2) Generate exactly one token in the provisioning path (per sandbox provision/reprovision), then:
   - Persist it via `SandboxInstanceRepository.rotate_bridge_token(sandbox.id, token, grace_seconds=30)`.
   - Inject the same token into `runtime_bridge_config["bridge"]["auth_token"]` (or equivalent spec-driven config field) that the provider uses to write `spec.runtime.config_path`.
3) Preserve grace-period semantics: if a sandbox already had a current token, it must move to previous with an expiry (30s), and the new token becomes current.
4) Keep fail-closed behavior: do not introduce any global token fallback or pairing-code usage; provisioning must not depend on `/pair`.

Recommended shape (choose one, but implement fully):
- Option A (preferred): Change `_generate_runtime_bridge_config(...)` to accept `bridge_auth_token: str` and never generate secrets internally.
- Option B: Change `_generate_runtime_bridge_config(...)` to return `(config, bridge_auth_token)` and plumb both through provisioning.

Important: keep mount isolation (config path must remain outside `/workspace/pack`) and keep using authoritative `gateway_url` persistence.
