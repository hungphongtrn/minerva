# Phase 03.5: Rewrite Orchestrator per ZeroClaw Gateway API - Research

**Researched:** 2026-03-07
**Domain:** ZeroClaw Gateway Integration, Control-Plane/Data-Plane Architecture, Sandbox Orchestration
**Confidence:** HIGH

## Summary

This phase requires a complete rewrite of Minerva's orchestrator to align with the ZeroClaw Gateway API reference and integration plan documents. The key architectural insight is the **control-plane/data-plane separation**:

- **Minerva (Control-Plane)**: Handles routing, sandbox lifecycle, policy enforcement, persistence, checkpoint/hydration orchestration
- **ZeroClaw Gateway (Data-Plane)**: Runs *inside each user sandbox* handling runtime ingress, tool execution, local workspace/memory semantics

**Core Tenancy Model**: 1 user → 1 sandbox. Multi-user → multi-sandbox. Multiple sessions for the same user reuse the same sandbox.

**Primary Recommendation**: Implement a spec-driven orchestrator where all gateway interactions are validated against `ZeroclawSpec`, with health-first fail-closed semantics, sandbox-scoped bearer token authentication, and proper event streaming bridge to Minerva's SSE envelope.

### Pattern 1: Health-First Fail-Closed Execution
**What**: Always poll `/health` before any execution; fail immediately if unhealthy
**When to use**: All gateway execution paths
**Example**:
```python
# Source: docs/zeroclaw-integration-plan.md Section 3A
async def execute_with_health_check(
    self, sandbox_url: str, token_bundle: GatewayTokenBundle
) -> GatewayResult:
    # Step 1: Health check (fail-closed)
    health = await self.poll_health(sandbox_url, token_bundle)
    if not health.healthy:
        return GatewayResult(
            success=False,
            error=GatewayError(
                error_type=GatewayErrorType.HEALTH_CHECK_FAILED,
                message=f"Health check failed: {health.status}",
                remediation="Sandbox may be unhealthy or unreachable. Try again later or reprovision.",
            ),
        )
    
    # Step 2: Only proceed if healthy
    return await self._execute_request(...)
```

### Pattern 2: Sandbox-Scoped Token Bundle with Grace Rotation
**What**: Per-sandbox tokens stored in DB with current + previous (grace period) support
**When to use**: All authenticated gateway calls
**Example**:
```python
# Source: docs/zeroclaw-integration-plan.md Section 4
@dataclass
class GatewayTokenBundle:
    current: str
    previous: Optional[str] = None
    previous_expires_at: Optional[datetime] = None

    def is_grace_token_valid(self) -> bool:
        if not self.previous or not self.previous_expires_at:
            return False
        return datetime.utcnow() < self.previous_expires_at

    def get_effective_token(self, attempt: int = 0) -> str:
        if attempt == 0:
            return self.current
        if self.is_grace_token_valid():
            return self.previous
        return self.current
```

### Pattern 3: Spec-Driven Configuration
**What**: All gateway paths, ports, auth modes loaded from `ZeroclawSpec`
**When to use**: All gateway service initialization and URL building
**Example**:
```python
# Source: src/integrations/zeroclaw/spec.py
class ZeroclawGatewayService:
    def __init__(self, spec: Optional[ZeroclawSpec] = None):
        self._spec = spec or load_zeroclaw_spec()
    
    def _get_health_url(self, sandbox_url: str) -> str:
        base = sandbox_url.rstrip("/")
        path = self._spec.gateway.health_path  # "/health"
        return urljoin(base + "/", path.lstrip("/"))
    
    def _requires_auth(self) -> bool:
        return self._spec.auth.mode == "bearer"
```

### Pattern 4: Event Streaming Bridge (Up → Minerva → Client)
**What**: Consume upstream ZeroClaw events (SSE/WS), translate to Minerva `OssSseEvent` types
**When to use**: When stream_mode is "sse" or "ws"
**Example**:
```python
# Source: docs/zeroclaw-integration-plan.md Section 6
async def execute_and_stream(
    sandbox_url: str,
    token_bundle: GatewayTokenBundle,
    request: dict,
) -> AsyncIterator[dict]:
    """Yield upstream runtime events as dicts."""
    # Consume upstream SSE/WS
    async for upstream_event in upstream_stream:
        yield map_upstream_to_oss_event(upstream_event)

# In runs.py
async for upstream in gateway.execute_and_stream(...):
    mapped = map_upstream_to_oss_event(upstream)
    if mapped:
        yield mapped.to_sse_lines()
```

### Pattern 5: Authoritative Gateway URL Enforcement
**What**: Only use explicitly stored `gateway_url` from DB; never construct synthetic URLs
**When to use**: All routing resolution
**Example**:
```python
# Source: docs/zeroclaw-integration-plan.md Section 7B
def _get_authoritative_sandbox_url(self, routing: RunRoutingResult) -> Optional[str]:
    # Only use stored gateway_url - no synthetic construction
    if routing.sandbox_url:
        return routing.sandbox_url
    
    # Fallback to lifecycle target
    if routing.lifecycle_target and routing.lifecycle_target.routing_result:
        sandbox = routing.lifecycle_target.routing_result.sandbox
        if sandbox and hasattr(sandbox, "gateway_url"):
            return sandbox.gateway_url
    
    # Fail-closed: return None instead of constructing URL
    return None
```

### Anti-Patterns to Avoid

1. **Calling `/pair` in production orchestration**: The pairing code flow is for human debugging only. Use orchestrator-managed tokens.

2. **Constructing gateway URLs from workspace IDs**: Always use the authoritative `gateway_url` stored during provisioning.

3. **Ignoring grace period on token rotation**: Without grace period, in-flight requests fail during rotation.

4. **Writing gateway config under `/workspace/pack`**: Config must be in writable workspace, not read-only pack mount.

5. **Synchronous health checks in request path**: Use async httpx with proper timeouts.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP client | aiohttp raw | httpx.AsyncClient | Better timeout/retry handling, FastAPI integration |
| Token generation | uuid4 or random | secrets.token_urlsafe | Cryptographically secure, URL-safe |
| Spec validation | Manual dict checks | Pydantic models | Type safety, validation errors, IDE support |
| SSE streaming | Manual string building | sse-starlette | Protocol compliance, edge case handling |
| Retry logic | Manual while loops | tenacity or async backoff | Exponential backoff, jitter, circuit breaker |
| JSON parsing | json.loads without error handling | Pydantic .model_validate() | Type coercion, validation, clear errors |

**Key insight**: The ZeroClaw Gateway API is well-documented with clear contracts. Build adapters to those contracts rather than abstracting them away.

## Common Pitfalls

### Pitfall 1: Health Check Timeout Cascade
**What goes wrong**: Health check timeout is too long, causing request queue buildup during gateway issues.
**Why it happens**: Default timeouts (30s+) don't account for fail-closed requirements.
**How to avoid**: Use short health timeouts (10s default), bounded retries (3), exponential backoff (max 5s).
**Warning signs**: Increasing queue depth, high latency during gateway issues.

### Pitfall 2: Token Rotation Race Condition
**What goes wrong**: New token deployed but in-flight requests with old token fail.
**Why it happens**: No grace period for token rotation.
**How to avoid**: Implement grace period (30s) where previous token is still accepted.
**Warning signs**: Intermittent 401s after sandbox reprovisioning.

### Pitfall 3: Pack Mount Isolation Violation
**What goes wrong**: Gateway config or runtime state written to `/workspace/pack` (read-only).
**Why it happens**: Confusing pack mount with workspace mount paths.
**How to avoid**: Validate config_path doesn't start with `/workspace/pack` in spec validator.
**Warning signs**: Permission errors during sandbox startup, read-only filesystem errors.

### Pitfall 4: Synthetic URL Construction
**What goes wrong**: Building gateway URLs from workspace IDs instead of using stored authoritative URL.
**Why it happens**: Provider doesn't return explicit gateway URL, or code tries to "optimize".
**How to avoid**: Enforce `gateway_url` storage in DB during provisioning; fail-closed if missing.
**Warning signs**: Connection errors to constructed URLs that don't match actual sandbox endpoints.

### Pitfall 5: Ignoring Stream Mode in Spec
**What goes wrong**: Code assumes streaming or non-streaming without checking `spec.gateway.stream_mode`.
**Why it happens**: Hardcoding behavior instead of being spec-driven.
**How to avoid**: Check `stream_mode` enum ("none", "sse", "ws") and branch accordingly.
**Warning signs**: Missing events, protocol errors when upstream changes stream support.

### Pitfall 6: Guest Mode Persistence Leak
**What goes wrong**: Guest runs accidentally create persistent state (checkpoints, sessions).
**Why it happens**: Forgetting guest checks in persistence paths.
**How to avoid**: Explicit `is_guest` guards before all persistence operations.
**Warning signs**: Database records with null workspace_id, unexpected checkpoint files.

```python
def resolve_gateway_tokens(
    self, sandbox_id: UUID
) -> Optional[GatewayTokenBundle]:
    """Resolve gateway tokens from sandbox metadata."""
    sandbox = self.get_by_id(sandbox_id)
    if not sandbox:
        return None
    
    # Get current and grace tokens from DB

    previous = sandbox.bridge_token_previous
    previous_expires_at = sandbox.bridge_token_previous_expires_at
    
    if not current:
        return None
    
    return GatewayTokenBundle(
        current=current,
        previous=previous,
        previous_expires_at=previous_expires_at,
    )
```

```python
# Source: docs/zeroclaw-integration-plan.md Section 6
def map_upstream_to_oss_event(upstream: dict) -> Optional[OssSseEvent]:
    """Map ZeroClaw upstream event to Minerva OSS event."""
    event_type = upstream.get("type")
    
    mapping = {
        "message.delta": OssEventType.MESSAGE,
        "message": OssEventType.MESSAGE,
        "tool.call": OssEventType.TOOL_CALL,
        "tool.result": OssEventType.TOOL_RESULT,
        "ui.patch": OssEventType.UI_PATCH,
        "state.update": OssEventType.STATE_UPDATE,
        "error": OssEventType.ERROR,
        "completed": OssEventType.COMPLETED,
        "failed": OssEventType.FAILED,
    }
    
    minerva_type = mapping.get(event_type)
    if not minerva_type:
        return None
    
    return OssSseEvent(
        id=upstream.get("id", str(uuid4())),
        type=minerva_type,
        ts=time.time(),
        run_id=upstream.get("run_id", "unknown"),
        data=upstream.get("data", {}),
    )
```

### Example 5: Bounded Recovery with Reprovisioning
```python
# Source: src/services/run_service.py pattern
async def _execute_with_recovery(
    self,
    routing: RunRoutingResult,
    message: str,
    session: Session,
) -> GatewayResult:
    """Execute with bounded recovery (max 3 attempts)."""
    MAX_RECOVERY_ATTEMPTS = 3
    last_error = None
    
    for attempt in range(MAX_RECOVERY_ATTEMPTS):
        # Get current authoritative URL
        sandbox_url = self._get_authoritative_sandbox_url(routing)
        token_bundle = self._resolve_gateway_tokens(routing, session)
        
        # Attempt execution
        result = await self._gateway_service.execute(
            sandbox_url=sandbox_url,
            message=message,
            token_bundle=token_bundle,
        )
        
        if result.success:
            return result
        
        # Check if recoverable
        if not self._is_recoverable_error(result.error):
            return result  # Fail fast
        
        last_error = result.error
        
        if attempt < MAX_RECOVERY_ATTEMPTS - 1:
            # Force reprovisioning
            routing = await self._recover_routing_target(routing, session)
            if not routing.success:
                break
    
    # Exhausted attempts
    return GatewayResult(
        success=False,
        error=GatewayError(
            error_type=GatewayErrorType.TRANSPORT_ERROR,
            message=f"Failed after {MAX_RECOVERY_ATTEMPTS} attempts",
        ),
    )
```

## Open Questions

1. **Upstream Event Schema**
   - What we know: ZeroClaw will emit events for message, tool_call, tool_result, etc.
   - What's unclear: Exact JSON schema for each event type
   - Recommendation: Implement flexible mapping with fallback; validate against schema when documented

2. **Session Continuity Semantics**
   - What we know: ZeroClaw uses `session_id` and `sender_id` in context
   - What's unclear: Which identifier is the canonical thread key for state restoration
   - Recommendation: Pass both; let ZeroClaw determine semantics

3. **Memory Persistence Paths**
   - What we know: Checkpoints should capture memory/session state
   - What's unclear: Exact file paths ZeroClaw uses for memory
   - Recommendation: Coordinate with ZeroClaw team on checkpoint contract

4. **Rate Limiting Coordination**
   - What we know: ZeroClaw has configurable rate limiting
   - What's unclear: Whether to enforce at Minerva, gateway, or both
   - Recommendation: Both layers; Minerva for coarse (per-user), gateway for fine (per-request)

5. **Stream Mode Transition**
   - What we know: Spec currently has `"stream_mode": "none"`
   - What's unclear: Timeline for SSE/WS support in ZeroClaw
   - Recommendation: Design for streaming now (AsyncIterator return types), implement when available
